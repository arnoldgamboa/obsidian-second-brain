Analysis of the main concern

The main question is, is the development for this epic taking too long? A few things to consider:

1 - We don’t have an objective way to tell. I was searching for a top level estimate on how long should the development take. I couldn’t find one. The only reference to timelines were those that refer to milestones and product’s target to launch the first venture by end of November. Those, however, are not objective top level development estimates. However…

1.1 - In my discussion with Eduardo, he believes that, give or take a few percentage, this project should take about 6 sprints. If we’re going to base on the current output, the estimates provided by William for the remaining tasks and factoring in the delays on dependencies, I can say that we roughly are around the vicinity of the expected development 

2 - Another point to consider are the dependencies. Beacon and the Figma design themes don’t match. So the team needs to “beaconize” the themes. The API also are not yet available, and therefore can be a bottleneck of development. 

3 - Lastly, we started slow. The new teams being on-boarded needed a few weeks to “heat up” and be speed up. It is expected that this will not be an issue moving forward. 

3.1 - Another thing is that when the team started this project, the solution pack is not ready yet. So we decided to start moving with the product specs being the guide and not the technical aspect. This made the progress “product driven” as opposed to what is ideally “engineering driven”. What is available for the team to work on based on what the product provided was the development guide and not the technical specification.

Where can we improve?

1 - Ideally, in every project, there should be a top level estimate on how long will it take to product. Smaller projects like, say game shuffle, or bigger ones like registration redesign, should be provided with a “guiding light” so that teams would work around that estimate.

2 - This suggestion came from Jomel, our Engineering Lead for Vigan. He proposes that if given a chance, ELs should be able to work with Staff Engineers even while working on the solution pack. This proposal provides potential benefits for the teams in the long run. This gives ELs a head start in the technical direction for a particular project and therefore, will have faster development cycle.
